# M5Dial-Badge 計画書

作成日: 2026-07-09
ステータス: 承認待ち

## 1. プロジェクトの目的

M5Dial を「電子バッヂ」として使えるガジェットを作る。

- 胸元や机上に置いた M5Dial の円形 LCD に、好きな画像・短い動画をループ表示する
- 表示コンテンツはスマホから差し替えられる。M5Dial 本体が Web アプリを提供し、LCD に表示する QR コードを読むだけでアクセスできる
- PC・専用アプリ・クラウド不要。スマホの標準カメラとブラウザだけで完結する

## 2. 背景と参考プロジェクト

`~/Documents/PlatformIO/Projects/splatoon`(AtomS3 Splatoon プレートプリンタ)で確立した以下の資産を流用する。

- SoftAP 起動 + Wi-Fi 設定 QR の LCD 表示(`M5.Display.qrcode`)
- キャプティブポータル(iOS / Android / Windows の接続性チェック URL ハンドリング)
- WebServer によるチャンクアップロードと LittleFS 保存
- ファームウェア埋め込みの単一 HTML による Web UI
- PlatformIO + Arduino + M5Unified の開発フロー、dist/ 配布物の構成

## 3. スコープ

### やること

- SoftAP + 2 段階 QR(Wi-Fi 参加 QR → URL QR)によるスマホ誘導
- STA モード(自宅 Wi-Fi 参加)対応のハイブリッド構成。STA 失敗時は AP に自動フォールバック
- Web UI からの画像アップロード(ブラウザ側で 240×240 に変換)
- Web UI からの動画アップロード(ブラウザ側で JPEG 連番=MJPEG 相当に変換)
- 複数コンテンツの保存とスライドショー連続再生
- ダイヤルによる送り/戻し、タッチによる一時停止、ボタン長押しで QR 表示
- Web UI での表示設定(順序・静止画表示秒数・動画ループ回数・明るさ)

### やらないこと(明示的に除外)

- H.264 / MP4 の端末上デコード(技術的に不可能)
- 音声再生(ブザーは効果音のみ)
- RFID・RTC の活用(ハードにはあるが本プロジェクトでは未使用)
- OTA アップデート(フラッシュ容量をコンテンツ保存に優先配分。将来検討)
- クラウド連携・複数バッヂ管理

## 4. 主要な技術判断(決定済み)

| 判断 | 決定 | 理由 |
| --- | --- | --- |
| Wi-Fi 方式 | ハイブリッド(SoftAP 主軸 + STA オプション) | イベント会場では SoftAP 直結が唯一確実(会場 Wi-Fi は AP isolation でスマホ→バッヂ通信不可)。自宅では STA でスマホのネット接続を維持。WLED / Tasmota と同じ業界標準構成 |
| スマホ誘導 | LCD の 2 段階 QR(Wi-Fi 参加 QR → URL QR) | キャプティブポータルの自動ポップアップは iOS / Android とも機種依存で信頼できない(調査ログ参照)。標準カメラで QR を 2 回読むだけの方式が最も確実 |
| 動画方式 | ブラウザ側で JPEG 連番に変換して転送 | ESP32-S3 に H.264 デコードは不可能。ブラウザの canvas / video API なら任意形式を受け付けられる |
| コンテンツ管理 | LittleFS に複数保存、スライドショー再生 | M5Dial に SD カードなし。LittleFS 約 5MB を確保 |
| 開発環境 | PlatformIO + Arduino + M5Dial ライブラリ(M5Unified ベース) | splatoon の資産・ノウハウを最大限流用 |
| プロビジョニング代替案 | SmartConfig / Improv BLE / DPP はすべて不採用 | SmartConfig: 専用アプリ必須+不安定。Improv BLE: BLE スタックの RAM 消費が PSRAM なし環境で危険。DPP: iOS・Arduino 非対応 |

調査の詳細は [log/2026-07-09-spec-discussion.md](../log/2026-07-09-spec-discussion.md) を参照。

## 5. 制約条件

- **ハードウェア**: M5Dial(M5StampS3 / ESP32-S3FN8)。Flash 8MB、**PSRAM なし**、SD カードなし
- **表示**: 円形 240×240 LCD(GC9A01)。四隅は物理的に見えない
- **保存容量**: LittleFS 約 5MB → 動画は合計 20〜40 秒程度(10fps・JPEG 品質による)が上限の目安
- **メモリ**: PSRAM なしのため、フレームバッファ・アップロードバッファ・JPEG デコードを内蔵 SRAM(約 512KB、実質利用可能はその一部)でやりくりする

## 6. マイルストーン

| フェーズ | 内容 | 完了条件 |

## 8. リスク

| リスク | 影響 | 対策 |
| --- | --- | --- |
| PSRAM なしによるメモリ不足 | 動画再生や大きいアップロードでクラッシュ | フレームは逐次デコード(全展開しない)、アップロードは小チャンク処理、BLE 等の大物スタックを載せない |
| 動画再生のフレームレート不足 | カクつき | 10fps を基準にし、実測で品質・解像度を調整。P4 の最初にベンチマークを行う |
| キャプティブポータルの機種依存 | 接続に迷うユーザー | 2 段階 QR を主経路とし、ポータルは URL 案内のみに割り切る |
| LittleFS 書き込み速度 | アップロードが遅い | チャンクサイズ調整。実測して Web UI に進捗表示 |
